- 과정명: AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 및 포트폴리오 제작 과정

- 사용 데이터: 교육용 샘플 데이터 사용
- 주의: 이 문서는 실제 AWS 내부 SOP가 아니라 교육용 샘플 SOP를 바탕으로 작성한 학습 자료입니다.

---

## 1. 내가 읽은 교육용 샘플 SOP 제목
SAMPLE-DCO-SOP-04: 샘플 Link Down 이벤트 기록 및 전달 절차

---

## 2. SOP의 목적
내가 이해한 목적: 교육용 샘플 로그에서 Link Down 이벤트가 관찰되었을 때, 발생 시간·관련 샘플 장비명·영향 범위·전달(Escalation) 필요 여부를 정리하는 연습을 하기 위한 문서입니다. 실제 장비를 점검하거나 네트워크를 진단하는 것이 아니라, 로그를 보고 "기록하고 정리하는 방식"을 익히는 것이 목적입니다.

---

## 3. SOP 절차 요약
1. 샘플 로그에서 Link Down 이벤트가 표시된 시간을 확인한다.
2. 이벤트에 표시된 샘플 장비명을 확인한다.
3. 같은 장비에서 반복 이벤트가 있는지 로그 안에서 확인한다.
4. Ticket opened 또는 Ticket escalated 표현이 함께 있는지 확인한다.
5. 영향 범위는 단정하지 않고, 로그에 나타난 관찰 내용만 정리한 뒤 혼자 판단하기 어려운 부분은 Escalation 필요 여부로 표시하고, Handover Memo나 Incident Report 초안 형태로 짧게 정리한다.

---

## 4. 주의사항
- 실제 장비에 접속하지 않습니다.
- 실제 네트워크 진단을 하지 않습니다.
- 실제 AWS 환경에 접속하지 않습니다.
- 실제 티켓 번호, 실제 장비 시리얼, 실제 IP를 사용하지 않습니다.
- AI 답변을 그대로 복사하지 않고 검토합니다.

내가 추가로 주의해야 한다고 생각한 점: AI가 "Handover Memo", "Incident Report" 같은 용어를 당연하다는 듯 사용할 수 있는데, 이런 용어의 정확한 뜻과 실제 사용 맥락은 공식 자료나 강의 자료로 별도 확인이 필요하다고 생각했습니다.

---

☑ 발생 시간을 확인했습니다.
☑ 샘플 장비명을 확인했습니다.
☑ 반복 여부를 확인했습니다.
☑ 관련 Ticket 표현을 확인했습니다.
☑ Escalation 필요 여부를 표시했습니다.
☑ 실제 정보가 포함되지 않았는지 확인했습니다.
☑ AI 답변에서 위험한 표현을 수정했습니다.

---

## 6. 내가 만든 SOP 체크리스트
□ 샘플 로그에서 Link Down 이벤트 발생 시간을 확인했다
□ 이벤트에 표시된 샘플 장비명(SAMPLE/DEMO/EDU 접두어)을 확인했다
□ 같은 장비에서 반복 이벤트가 있는지 확인했다
□ Ticket opened 또는 Ticket escalated 표현이 함께 있는지 확인했다
□ 영향 범위를 단정하지 않고 로그에 나타난 관찰 내용만 정리했다

---

## 7. 오늘 사용한 AI 프롬프트 요약
역할: 긴 SOP 문서를 학생이 따라 읽기 쉬운 체크리스트로 바꾸는 학습 도우미 / SOP 분석 내용이 교육용으로 적절한지 검토하는 학습 도우미

목표: 교육용 샘플 SOP를 목적·절차·주의사항·확인항목으로 정리하고, Notion 체크리스트 형태로 변환하고, 최종적으로 교육용 실습자료로서 적절한지 검토받기

상황: AWS DCO 인턴십 직무 이해를 위한 생성형 AI 활용 수업 수강 중이며, 입력한 SOP는 실제 AWS 내부 SOP가 아닌 교육용 샘플 SOP임을 명시

입력 자료: SAMPLE-DCO-SOP-04 (샘플 Link Down 이벤트 기록 및 전달 절차) 전문 — 사용 목적, 적용 범위, 절차 7단계, 주의사항, 확인항목 포함

출력 형식: (1) SOP 목적/절차 요약/주의사항/확인항목/한 문장 요약, (2) Notion 체크리스트(□ 형식), (3) 검토 결과(수정 필요 표현/수정 이유/더 안전한 표현/학생이 확인할 사항)

제한 조건: 실제 AWS 내부 절차처럼 단정하지 않기 / 실제 장비 점검·네트워크 진단·AWS 환경 접속 요구 금지 / 실제 티켓 번호·장비 시리얼·IP·계정정보 금지 / 원인을 하나로 단정하지 않기 / AI 답변은 초안이므로 학생이 검토할 부분 표시

---

## 8. AI 답변을 보고 내가 수정한 점
실제 절차처럼 보여 수정한 표현: 특별히 크게 수정할 부분은 없었으나, "Handover Memo", "Incident Report" 같은 용어가 실제 내부 문서 양식처럼 보이지 않도록 "교육용 샘플" 맥락을 계속 유지하며 읽었습니다.

실제 장비 점검처럼 보여 삭제한 표현: 절차 자체에 실제 장비 접속이나 진단 명령을 요구하는 표현은 없어서 별도로 삭제한 부분은 없었습니다.

원인을 단정해서 바꾼 표현: "영향 범위를 단정하지 말고 관찰 내용만 정리한다"는 원칙이 이미 문서에 명시되어 있어, 이 부분을 검토할 때도 원인을 하나로 단정하지 않도록 계속 확인했습니다.

어려워서 쉽게 바꾼 표현: "Handover Memo" → "다음 담당자에게 넘길 간단한 인수인계 메모", "Incident Report 초안" → "사건 기록 보고서의 첫 번째 버전"처럼 풀어서 이해했습니다.

---

## 9. 오늘의 느낀 점
SOP를 읽고 새롭게 알게 된 점은: SOP가 단순히 "이렇게 하라"는 지시문이 아니라, 원인을 함부로 단정하지 않고 사실만 기록한 뒤 필요할 때 전달(Escalation)하는 흐름으로 설계되어 있다는 것을 알게 되었습니다.

SOP를 체크리스트로 바꾸면 좋은 점은: 긴 절차를 한눈에 확인하고 빠뜨리지 않고 따라갈 수 있어서, 실제로 기록을 남길 때 실수를 줄이는 데 도움이 될 것 같습니다.

AI 답변을 사용할 때 조심해야 할 점은: AI가 만든 검토 결과나 체크리스트도 그대로 믿기보다, 실제로 절차를 따라가 보면서 빠진 부분은 없는지 스스로 다시 확인하는 태도가 필요하다는 것을 느꼈습니다.

다음 차시에서 준비해야 할 것은: Ticket 작성이나 Escalation 기준처럼, 오늘 다룬 SOP와 이어지는 실제 업무 흐름 개념을 더 살펴보고 싶습니다.
